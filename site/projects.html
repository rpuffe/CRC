Flightdeck

An opinionated AWS delivery path for turning a container and a small YAML manifest into a secure, inspectable HTTPS deployment—without requiring every application repository to reinvent infrastructure and CI/CD.

## the problem

AI-assisted tools can produce useful application code quickly, but getting that code into AWS still requires decisions about identity, networking, state, security scanning, promotion, rollback, and cost. Flightdeck moves those decisions into a narrow platform contract so the application supplies only what the platform genuinely needs.

small application contract

A Dockerfile, health endpoint, and versioned manifest replace bespoke infrastructure in every application repository.

credential-free delivery

GitHub Actions exchanges repository identity for short-lived, per-application AWS credentials through OIDC.

immutable promotion

Production receives the exact image digest tested in development instead of rebuilding from the same source tag.

cost-aware operation

Services can scale to zero overnight while shared networking and routing keep the platform economical to demonstrate.

## architecture

Account-level infrastructure establishes the shared network, routing, image repositories, OIDC provider, and deployment roles. Each registered application gets an isolated Terraform state path and a scoped stack for its ECS service, task definition, logging, alarms, and DNS route.

## engineering controls

identity and blast radius

  • Exact-repository OIDC trust instead of owner-wide access
  • Per-application deployment roles and permissions boundaries
  • Application and environment state-path isolation
  • Private ECS tasks behind a shared HTTPS entry point

delivery integrity

  • Schema and preflight validation before AWS mutation
  • Trivy and Checkov gates for high and critical findings
  • Saved Terraform plans for reviewed apply behavior
  • Build-once image promotion with digest verification

## decisions and tradeoffs

ECS instead of Kubernetes

The target workload is a small HTTP container. ECS provides the scheduling and AWS integration needed without making cluster administration part of the application contract.

contract before portal

A versioned YAML manifest and reusable workflow test the platform boundary directly. A developer portal would add another interface before the underlying contract is proven.

shared foundations, isolated apps

Networking and routing are shared to control cost; deployment roles, state, services, task definitions, storage, and logs remain scoped by application and environment.

cost before high availability

This reference platform intentionally favors low idle cost and inspectability. Single-task services and scale-to-zero are useful here, but they are not a high-availability claim.

## evidence, not just architecture

6m 38s fastest cold run

The measured path started with a plain-language application specification and ended at a public HTTPS URL, including repository creation, build, infrastructure, and deployment.

multiple application shapes

The same platform contract has delivered several independent applications, exposing reusable-platform problems that a single demonstration would not reveal.

failure-driven hardening

Real deployments found missing replacement and cleanup permissions, unsafe trust breadth, image-lifecycle risk, and scaler failure behavior. Those became tests and narrower controls.

documented residual risk

The specification and threat model state what the platform protects, what it deliberately does not solve, and which operational risks remain instead of hiding them behind a production-ready label.

The strongest result is not the number of AWS services. It is the repeatable contract: measure the path, capture failures, and improve the control plane instead of pushing infrastructure complexity into every application.

## current boundaries and next proof

Flightdeck is an inspectable reference platform, not a claim that every workload belongs on this architecture. Its current contract is deliberately narrow: one HTTP container per service, one AWS region, cost-first task counts, and limited day-two automation.

  • Run the full disposable lifecycle under the constrained per-app deployment role
  • Add continuous synthetic monitoring with explicit service objectives
  • Detect Terraform drift without creating another apply path
  • Rehearse state and application recovery against disposable resources
  • Publish a sanitized, durable evidence bundle that works while services sleep
